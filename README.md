# RAG System

This project is a lightweight Retrieval-Augmented Generation (RAG) pipeline for answering questions from company policy data stored in a local Chroma vector database.

## Project Overview

The workspace contains a simple end-to-end RAG workflow:

1. Load policy documents from the `docs/` folder.
2. Split the text into smaller chunks.
3. Generate embeddings and store them in ChromaDB.
4. Retrieve the most relevant chunks for a user question.
5. Generate a grounded answer using the retrieved context.

All of the top-level scripts below are thin CLI wrappers: each one adds
`Utils/` to `sys.path`, imports the shared `script.py` module, and calls a
single function in it to run that stage of the pipeline. `script.py` holds
the actual splitting, ingestion, retrieval, and generation logic.

## Folder Structure

- `Utils/script.py`
  Core module containing the real pipeline logic (`run_ingestion`,
  `run_retrieval`, `run_multi_query_retrieval`, `answer_query`,
  `get_vector_store`, `get_chat_model`, `history_aware_answer`,
  `run_history_chat_loop`, `run_splitter_demo`). Every other script below
  is a wrapper around this file.

- `ingestion_pipeline.py`
  Builds the vector store from the policy documents. Calls `script.run_ingestion()`.

- `retrieval_pipeline.py`
  Retrieves relevant document chunks for a supplied question. Calls `script.run_retrieval(query)`.

- `answer_generation.py`
  Retrieves context and generates a final answer. Calls `script.answer_query(query)`.

- `history_aware_generation.py`
  Supports a simple multi-turn chat-style workflow using conversation history.
  Calls `script.history_aware_answer(...)` for a one-off query, or
  `script.run_history_chat_loop()` for an interactive session.

- `multi_query_retrieval.py`
  Generates multiple query variations, retrieves results for each, and
  produces a final answer. Calls `script.run_multi_query_retrieval(query)`.

- `recursive_character_text_spliiter.py`
  Demonstrates how `CharacterTextSplitter` and `RecursiveCharacterTextSplitter`
  behave on sample text. Calls `script.run_splitter_demo()`.
  *(Note: filename has a typo — "spliiter" instead of "splitter".)*

- `run_all_rag.py`
  Interactive loop that runs the full ingestion-to-answer flow end-to-end via
  `script.run_all_flows(question)`. Also prints debug output on startup
  confirming whether `.env` was found and whether `OPENROUTER_API_KEY`
  loaded successfully — useful for diagnosing environment issues.

- `real_rag.py`
  A consolidated module that re-exposes `run_ingestion`, `run_retrieval`,
  and `run_all_flows` as a cleaner programmatic API on top of `script.py`.
  Overlaps in purpose with `run_all_rag.py`; the two could likely be merged.

- `tempCodeRunnerFile.py`
  Leftover scratch file auto-generated by the VS Code "Code Runner"
  extension. Not part of the actual pipeline — safe to delete or
  add to `.gitignore`.

- `docs/policy_data.txt`
  Source policy dataset used for retrieval.

- `db/chroma_db/`
  Persisted Chroma vector store created during ingestion.

## Query Argument Handling

`retrieval_pipeline.py`, `answer_generation.py`, `multi_query_retrieval.py`,
and `history_aware_generation.py` each filter `sys.argv` through a
`looks_like_runner_command()` check before treating it as a question. This
strips out shell/IDE noise (e.g. `python.exe`, `cd /d`, `&&`, `-u`, `--`)
that can get passed in when running scripts via VS Code's "Run Python File"
button, so that noise doesn't get sent to the model as if it were the
user's question. If no valid query survives filtering, the script falls
back to an interactive `input()` prompt.

## Data Description

The source dataset in `docs/policy_data.txt` contains policy records grouped into categories such as:

- Remote Work & Hours
- Health & Wellness
- Office & Security
- Expense & Travel
- Time Off & Leave
- IT & Hardware
- Benefits & Perks
- Performance & Growth

It also includes reference codes such as `OX-1001`, `OX-1002`, and similar policy IDs that are used to trace specific policy statements.

## Environment Setup

The project expects an environment variable named `OPENROUTER_API_KEY`.

You can place it in a `.env` file in the project root or inside the `venv` directory.

Example:

```env
OPENROUTER_API_KEY=your_api_key_here
```

`run_all_rag.py` explicitly resolves and prints the expected `.env` path on
startup, and confirms whether `OPENROUTER_API_KEY` was loaded — handy for
debugging a missing or misplaced `.env` file.

## Running the Pipeline

From the project root, use the project virtual environment Python executable.

### 1. Build or load the vector store

```powershell
Set-Location "c:\Users\my\Desktop\RAG SYSTEM"
& "c:/Users/my/Desktop/RAG SYSTEM/venv/Scripts/python.exe" ingestion_pipeline.py
```

### 2. Run direct retrieval

```powershell
Set-Location "c:\Users\my\Desktop\RAG SYSTEM"
& "c:/Users/my/Desktop/RAG SYSTEM/venv/Scripts/python.exe" retrieval_pipeline.py "What are the core collaboration hours for all teams as defined in reference code"
```

### 3. Generate a grounded answer

```powershell
Set-Location "c:\Users\my\Desktop\RAG SYSTEM"
& "c:/Users/my/Desktop/RAG SYSTEM/venv/Scripts/python.exe" answer_generation.py "What are the core collaboration hours for all teams as defined in reference code"
```

### 4. Use the history-aware chat flow

```powershell
Set-Location "c:\Users\my\Desktop\RAG SYSTEM"
& "c:/Users/my/Desktop/RAG SYSTEM/venv/Scripts/python.exe" history_aware_generation.py "What are the core collaboration hours for all teams as defined in reference code"
```

### 5. Use the multi-query retrieval flow

```powershell
Set-Location "c:\Users\my\Desktop\RAG SYSTEM"
& "c:/Users/my/Desktop/RAG SYSTEM/venv/Scripts/python.exe" multi_query_retrieval.py
```

If a question is not passed as a command-line argument, the script will prompt for one interactively.

### 6. Run the full flow interactively

```powershell
Set-Location "c:\Users\my\Desktop\RAG SYSTEM"
& "c:/Users/my/Desktop/RAG SYSTEM/venv/Scripts/python.exe" run_all_rag.py
```

Prompts for questions in a loop (type `quit` to exit) and runs the complete
ingestion → retrieval → answer flow via `script.run_all_flows()`.

## Notes

- The retrieval workflow depends on the persisted Chroma index in `db/chroma_db/`.
- If the database is missing or empty, run `ingestion_pipeline.py` first.
- The source document file contains policy data in a semi-structured layout, so chunking and retrieval may return multiple policy fragments related to the same topic.
- `real_rag.py` and `run_all_rag.py` currently overlap in responsibility — consider consolidating them.
- `tempCodeRunnerFile.py` is IDE-generated clutter, not a real pipeline component.
